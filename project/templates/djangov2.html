Welcome to the Django Section Part2


Aim of this project: To build a production standard version of the web staffing app I built pervious. This would be done be using and learning about docker for me production environment, using PostgreSQL as the datebases, scraping the available shifts from a portal using the beautiful soup and asynchronous task manager with Celery with all the code being well tested. The app takes user inputs as their availability to work. Scraps the portal to update available shifts, then using an algorithm to match available shifts to users’ skills and availability.


I created a new Django project, rather than using virtual env I used pipenv to track the dependencies within the piplock file. Following this I created an image using docker to act as my development environment. I added a .gitignore file I used this one from github .gitignore Repository. I then added the docker-compose.yml to hold my environment variables such as debug, secret key and allowed hosts and added that to the gitignore file Once the .gitignore file was added I created a Github repository to store the code as I was able to work on it from both my main desktop and laptop when commuting. GitHub Repository Once I had created the environment to develop in, I created a custom user model as now recommended.


To make my log in and log functionality I used Django All Auth to create a robust authentication system. I added the ability to reset passwords via an email link. The email backend is set up using Twilio ftp for simplicity but I’m planning to change to the Gmail interface on going foward. User App GitHub


After this it was time to start adding the functionality of the website. I created a model with a form to take the time and date of each worker is available to work.
I also added a html calendar to display their availability/booked shifts in a readable format. The user is able to click on text, which will take them through to the object edit page or if the shift is booked the detail page. To guide me through adding this I used this guide HTML Calendar Guide
I plan on adding two email conformations here.
1. Every time availability is added to send a confirmation of the updated availability
2. Every time the object moves from available to booked confirmation of the shift’s details and a conformation link to confirm they have seen the shift has been booked.
Calendar App GitHub


The next functionality was to add the shifts model for this I needed more information from the users such as which hospitals and area they can work. I updated the custom user model to include a hospital field and area field. I added this to using a manytomany field and created models for each field. I then added the fields to the sign up form and to make it easier for the user to select their options I added the FilteredSelectMultiple widget which I liked from the Django admin panel. I found a useful Stack Overflow post hereon how to implement them. Although this created a small issue with the field header for the following field not rendering in the correct place and this still needs to be resolved. Sign Up Template GitHub | User App GitHub


Once I had the correct information being captured by the sign-up process, I finished the shift model which accepted start time, end time, hospital and area. I added a simple form to accept this information. Although the plan is to not use this method and have the shift info scraped from the customers website and automatically updated. Shifts App Github


Following this I then wrote an algorithm to compare all available shifts and user availability and automatically books the user onto the shift whilst updating the shift info to the user. Created this is triggered using the post_save signal when a shift is saved. But this will be added to a celery task and will run periodically to avoid slowing the website down for the users. Sign Up Template GitHub


At this point the app may look similar to the previous app, there have been many improvements to the structure and scalability of the app. Such as it is using a production standard database. One of the biggest improvements is the tests I have written to check the website is functioning, this will make it much easier to debug as the app grows.
